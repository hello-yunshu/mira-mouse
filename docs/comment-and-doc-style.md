# 备注与文档约定

本文件约定 Mira 项目的代码备注（注释）与文档撰写规范，目标是保持代码简洁、文档清晰可维护。

## 一、代码备注原则

**核心准则：备注应解释"为什么"，而非"做什么"。代码本身已经说明了"做什么"。**

### 1.1 不添加备注的情况

以下备注属于冗余，**禁止新增**，审核时应删除：

- **复述代码行为**：在显而易见的代码上方重复描述，例如 `// 增加计数器` 在 `counter += 1` 之前
- **解释语言语法**：如 `// 导入模块`、`// 定义结构体`、`// 创建向量`
- **复述函数名/类型名**：函数名 `parse_dpi` 上方写 `// 解析 DPI`
- **空洞描述**：如 `// 返回结果`、`// 初始化`、`// 处理事件`
- **与文档注释重复**：`///` 文档注释已说明的项，不再在行内重复一遍
- **无信息分隔线**：如 `// ----- helpers -----`（除非在大段代码中确有导航价值）
- **被注释掉的代码**：应直接删除，如需保留请用 git 历史而非注释
- **测试步骤描述**：测试代码应自解释，不在 `render()` 上方写 `// 渲染组件`

### 1.2 应当保留/添加备注的情况

以下备注属于必要，**应当保留**，新增代码时也按此标准判断：

- **解释 WHY**：业务逻辑决策、变通方案、bug 修复背景（推荐附 issue 编号，如 `// 修复 #11：导入逐字段写入`）
- **协议/算法细节**：HID 协议命令（0xD6 等）、非显而易见的算法步骤、外部规范引用
- **契约说明**：UI↔Plugin 契约（mutationDecl、rangeSpec、capability 契约等）、跨模块接口约定
- **安全性不变量**：`unsafe` 块必须注释说明为何该操作是安全的
- **锁纪律/并发**：锁的获取顺序、死锁规避策略、不可重入锁的注意事项
- **跨平台差异**：macOS / Windows / Linux 行为不同的实现说明
- **项目硬约束**：版本号规则、协议命令、构建溯源、签名公钥等项目级约束
- **i18n key 解释**：如 `// mock.mouseLabel` 标注后端返回的 i18n key
- **未解决的 TODO**：标注清晰且仍有意义的 `// TODO`（附 issue 编号更佳）
- **公共 API 文档注释**：Rust `///`、TypeScript JSDoc 用于公共 API

### 1.3 备注语言

- 代码备注使用**中文**为主
- 涉及专有名词、协议命令、技术术语时可保留英文
- SPDX 许可证头保持原样

## 二、文档撰写规范

### 2.1 语言要求

- **中文为主，英文为辅**：所有 `.md` 文档以中文撰写
- `README.en.md` 为英文版例外，保留全英文
- 代码块、命令、配置项、技术术语保留英文原词
- 长文档可在关键术语后括注英文，如"签名密钥（signing key）"

### 2.2 文档维护要求

- **内容必须与项目当前状态一致**：禁止保留"未配置"、"尚未发布"等过时表述
- **版本号同步**：CITATION.cff、CHANGELOG、README 中的版本号必须与最新发布一致
- **链接有效性**：文档内引用的 `docs/*.md` 链接必须指向真实存在的文件
- **避免模板套话**：CONTRIBUTING、CODE_OF_CONDUCT 等应有项目实质内容，不照搬空模板

### 2.3 文档结构

- 每个文档开头有简短标题与一句话说明其用途
- 长文档使用二级/三级标题分节
- 命令行操作用代码块包裹并标注语言（```bash 等）
- 涉及多步骤流程用有序列表

## 三、审核清单

定期审核备注与文档时，按以下清单检查：

| 检查项 | 判定 |
|---|---|
| 备注是否解释 WHY 而非 WHAT | 删除复述 WHAT 的备注 |
| 是否有被注释掉的代码 | 删除，用 git 历史保留 |
| TODO 是否已解决 | 已解决则删除，未解决则保留 |
| 文档版本号是否最新 | 同步到当前发布版本 |
| 文档链接是否有效 | 修复断链或移除引用 |
| 文档是否中文为主 | 翻译英文文档为中文为主 |
| 文档内容是否与现状一致 | 更新过时表述 |
