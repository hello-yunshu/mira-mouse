# Mira Mouse 日志系统实现提示词

> 将本文件完整交给 AI IDE 执行。目标是直接在 `hello-yunshu/mira-mouse` 当前代码库中完成实现、测试与验证，而不是只输出设计建议。

## 任务角色

你是一名负责桌面端工程、Rust/Tauri、React/TypeScript、可观测性、隐私保护和交互设计的高级工程师。请在现有 Mira Mouse 项目中实现一套可靠、易用、可维护的本地日志与诊断系统。

仓库：`https://github.com/hello-yunshu/mira-mouse`

请基于仓库当前 `main` 分支的真实代码工作。不要仅凭本提示词假设文件结构、依赖版本或既有实现；开始修改前必须先检查仓库现状。

6. 不要为了实现日志而大范围重构无关模块，但应避免继续把大量代码堆入已经过大的文件。

## 二、核心目标

实现统一的本地日志系统，至少覆盖：

- Mira 程序与 Tauri/Rust 后端日志
- React 前端中经过主动记录的重要日志
- 鼠标插件运行日志
- 本地 AI runtime、模型、handler、预测与回退日志
- 应用启动、关闭、更新、设备连接、插件加载和关键错误

用户能够在 Mira 应用内部查看、筛选、搜索、暂停显示、自动跟随、复制、导出和删除日志，并能生成脱敏诊断包。

日志系统必须满足：

- 本地优先，不上传遥测
- 默认开销低
- 结构化、可筛选
- 有界内存与磁盘占用
- 不泄露敏感信息
- 不影响鼠标控制和本地 AI 主流程
- 出现日志系统故障时不能拖垮应用

## 三、不可破坏的约束

### 1. 主窗口

- 不修改 Mira 主窗口的默认宽度、高度、最小尺寸和可最大化策略。
- 不以扩大主窗口作为容纳日志界面的手段。
- 日志的默认查看方式必须位于 Mira 主应用内部。
- 不将独立 Tauri 子窗口作为默认入口。
- 只有在主功能全部完成、代码可复用且不会明显增加复杂度时，才可以额外提供“在独立窗口打开”；否则不要添加未完成入口，只保留可扩展结构。

### 2. 关于页入口

- 在“关于”页面中新增独立的“日志与诊断”卡片。
- 该卡片必须是关于页内容区的第二块卡片，即位于应用品牌/说明卡片之后、当前版本与构建信息卡片之前。
- 原有版本、插件、更新、许可证、隐私和反馈内容继续保留，只按需要顺延位置。
- 卡片文案简洁，说明日志仅在本地保存，可查看程序、插件和本地 AI 的运行记录。
- 卡片保留一个明确主操作，例如“打开日志与诊断”。不要在入口卡片堆放大量操作。

### 3. 界面与产品一致性

- 复用 Mira 当前的字体、圆角、间距、按钮、卡片、背景、毛玻璃效果、明暗主题和动效规则。
- 不引入与现有视觉语言明显冲突的通用后台管理界面或终端风格界面。
- 不把每条日志设计成厚重的大卡片。
- 新增中文和英文翻译，不在组件内硬编码面向用户的文本。
- 新文件遵守项目许可证头、目录边界、代码风格和质量检查规则。

### 4. 插件边界

- 先理解当前插件是否为声明式、签名包以及宿主如何执行插件工作流。
- 不改变插件的安全模型，不为了日志引入可执行插件代码。
- “插件日志”主要记录宿主在加载、验证、匹配、解析和执行插件工作流时产生的结构化事件。
- 不要求现有已签名插件重新打包，除非仓库已有兼容的可选日志字段且可以保持向后兼容。

### 5. 隐私与安全

默认不得记录或导出：

- 完整用户主目录和用户名
- 系统令牌、私钥、公钥私密材料、签名秘密、认证信息
- 未经处理的设备序列号和可稳定追踪用户的标识
- 原始 HID 数据包、完整特征报告或高频二进制内容
- 完整电量历史或其他无关用户数据
- 本地 AI 的完整输入历史
- 可能包含敏感内容的任意大段文件内容

确需诊断时，只允许在用户主动开启的短时诊断模式中记录有限、脱敏、限量的数据，并自动恢复默认等级。

## 四、日志架构要求

### 1. 单一日志入口

建立统一日志门面，避免同时维护互不连通的多套日志系统。

如果仓库尚无正式日志框架，请比较适合当前项目的实现方式，例如：

- Rust `tracing` 生态配合自定义内存层、文件层和前端事件层
- Tauri 日志插件配合必要的结构化日志服务
- 其他足以满足本任务需求的轻量方案

选择时以以下能力为准，而不是盲目增加依赖：

- 结构化字段
- Rust 模块级 target
- 滚动文件
- 内存环形缓冲
- 实时推送给前端
- 运行时筛选或临时提高采集等级
- 脱敏
- 可测试性

不得出现一套写文件、另一套供 UI、第三套供诊断导出的重复实现。文件、UI 和诊断包应基于同一日志服务或同一标准化事件流。

### 2. 统一数据模型

实现稳定、可序列化的日志结构。可根据 Rust/TypeScript 习惯调整名称，但语义至少包含：

```ts
type LogLevel = 'error' | 'warn' | 'info' | 'debug' | 'trace';

type LogSource = 'app' | 'frontend' | 'plugin' | 'local-ai';

type LogEntry = {
  id: number | string;
  timestamp: string;
  level: LogLevel;
  source: LogSource;
  target: string;
  message: string;
  sessionId: string;
  correlationId?: string;
  fields?: Record<string, string | number | boolean | null>;
};
```

要求：

- 时间使用明确时区的标准格式，UI 可本地化显示。
- `message` 面向开发者但应尽量可读。
- 可变技术信息放在 `fields`，不要把所有数据拼接成不可解析字符串。
- `target` 表示具体模块或子系统，如设备发现、插件验证、AI 预测、更新器等。
- 每次应用启动生成新的 `sessionId`。
- 对一次跨模块操作尽可能使用 `correlationId` 串联，例如：界面操作 → Tauri command → 插件工作流 → HID 写入 → 校验结果。
- 字段必须可安全序列化；禁止无限递归、大对象和未限制字符串。

### 3. 日志等级语义

统一等级含义：

- `error`：操作失败、不可恢复异常、关键组件启动失败
- `warn`：已回退、已重试、状态异常但应用仍能继续
- `info`：低频生命周期、用户触发的关键操作结果、版本与状态变化
- `debug`：诊断所需的流程、耗时、分支选择和结构化上下文
- `trace`：极低层且高频的细节，仅短时诊断使用

不得把高频轮询、每次电量采样、每个 HID 调用都写为 `info`。默认运行时应保持安静。

### 4. 内存、查询与实时事件

实现有界内存缓冲，建议默认保留最近约 3,000～5,000 条，具体值集中配置并有测试。

后端需要提供语义等价的能力：

- 查询历史日志，支持来源、最低等级、关键字、会话和时间范围
- 分页、游标或明确的数量上限，禁止一次通过 IPC 返回无限日志
- 获取日志状态，例如当前采集等级、文件占用、会话标识和计数
- 实时向前端推送新日志
- 删除磁盘历史日志
- 导出日志或诊断包
- 临时调整采集等级
- 接收少量经过筛选的前端日志

命令和事件名称可遵循项目现有命名规范，不必机械采用某个固定名称。

实时推送必须：

- 非阻塞或尽量不阻塞产生日志的业务线程
- 在前端未打开日志页时仍能正常记录，但不应造成无意义 UI 事件压力
- 对高频重复事件进行限流、聚合或降级
- 防止日志订阅形成自我递归日志

### 5. 持久化和轮转

日志保存在系统标准应用日志或应用数据目录内，不写入源码目录和当前工作目录。

提供合理默认值，并将值集中到可维护配置中。建议基线：

- 默认采集 `info` 及以上
- 单文件约 5 MB 后轮转
- 总磁盘占用约 20 MB 上限
- 最多保留 7～14 天
- 超限优先删除最旧日志
- `debug`、`trace` 默认不长期保存

可根据所选框架的可靠能力微调，但必须同时具备按时间或空间自动清理机制。用户不应依赖手动删除来维持应用正常运行。

文件写入失败、目录无权限或磁盘不足时，日志系统应降级为内存日志，并以非递归方式向用户暴露状态。

### 6. 脱敏层

在进入持久化、前端显示和导出前使用统一脱敏策略，不要只在导出时临时替换。

至少处理：

- 主目录路径替换为通用标记
- 用户名和系统账户信息
- 设备序列号、稳定 ID、地址类标识
- URL 中的查询参数和凭据
- key、token、secret、authorization、cookie 等常见字段
- 超长字符串截断
- 控制字符和不可显示内容

对字段名使用允许列表与拒绝列表结合的策略。脱敏逻辑必须有单元测试。

## 五、应接入的日志来源

### 1. 程序与后端

至少覆盖低频关键事件：

- 应用启动完成、版本、平台、会话开始
- 配置加载失败与回退
- 窗口、托盘、通知和更新器的关键失败
- 设备发现、连接、断开、刷新和选择结果
- Tauri command 的重要失败
- 后台任务异常退出
- 应用关闭或异常终止前可捕获的信息

将当前散落的临时后端输出逐步迁移到统一日志。至少迁移本次功能涉及的启动、窗口、托盘、更新、插件和本地 AI 关键路径。不要为追求“零临时输出”而无边界重写整个项目。

### 2. 前端

只记录经过设计的重要事件与异常，不要自动劫持所有 `console.log`。

可记录：

- 页面或关键功能加载失败
- Tauri invoke 失败
- 用户触发的重要操作结果
- 导出、删除、插件更新等失败
- React 可捕获的严重运行异常

要求：

- 使用小型前端日志适配器，统一字段和等级。
- 发送到后端时限制频率、单条长度和字段大小。
- 不能因后端日志命令失败而不断重试或递归记录。
- 开发环境仍可保留合理的控制台体验。

### 3. 插件运行时

记录宿主侧插件事件，例如：

- 插件包发现、读取和版本选择
- 签名校验和信任结果
- bundled 与 installed 版本决策
- 设备匹配及匹配失败原因摘要
- 工作流、mutation、parser 或 capability 执行结果
- 操作耗时、重试、回滚和错误类别
- 插件更新、替换、拒绝和恢复

建议字段：

- `pluginId`
- `pluginVersion`
- `deviceFamily` 或非敏感设备类别
- `workflowId` / `mutationId`
- `operation`
- `durationMs`
- `result`
- `errorCode`

默认不记录原始 HID payload。需要低层诊断时，仅在临时 `trace` 会话中记录经过裁剪和脱敏的摘要，并设置严格数量上限。

### 4. 本地 AI

至少记录：

- runtime 可用性和版本
- 模型包、handler 包的选择和签名校验结果
- runtime 启动、停止、重启和握手结果
- 预测请求编号、批次数、样本数量摘要和耗时
- 本地 AI 结果是否有效
- 回退到确定性算法的原因
- 超时、stdout 关闭、协议解析失败和子进程异常退出
- 崩溃计数、冷却、自动恢复和 bundle 回滚结果

检查当前 runtime 子进程的 `stderr` 处理。如果目前只读取或保留尾部但未被消费，应将其安全接入统一日志：

- 按行或合理块读取，避免阻塞子进程
- 每行限制长度
- 对重复内容限流或折叠
- 根据内容和上下文映射到适当等级
- 默认不把普通 stderr 全部提升为错误
- 子进程退出时保留有限尾部用于失败诊断
- 不因日志读取线程造成死锁、资源泄漏或无法回收

## 六、应用内日志页面

### 1. 导航方式

点击关于页第二块“日志与诊断”后，在当前 Mira 主窗口内部打开完整日志页面。

优先采用与现有状态管理和页面切换方式一致的实现，例如：

- 关于页内部的子页面状态
- App 级新增嵌套 view
- 项目已有的轻量导航模式

选择侵入最小、返回逻辑清晰、测试容易的方式。不要为一个页面引入大型路由框架。

日志页必须能返回关于页，并保持主窗口原尺寸。

### 2. 信息结构

在当前窄窗口中采用垂直布局，至少包含：

1. 顶部标题和返回操作
2. 日志来源筛选
3. 搜索框
4. 最低日志等级筛选
5. 自动跟随开关
6. 暂停/继续界面更新
7. 日志列表
8. 状态与操作区

来源筛选默认提供：

- 全部
- 程序
- 插件
- 本地 AI

前端日志可以归入“程序”，同时在详情中显示 `source=frontend`；若实际信息量足够，也可单独提供“前端”，但不要让默认界面过度复杂。

等级筛选优先采用“最低等级”模型：

- Error
- Warn 及以上
- Info 及以上
- Debug 及以上
- Trace

默认显示 `Info 及以上`。

### 3. 日志列表

每条日志默认显示：

- 本地化时间
- 等级
- 来源
- 一行或两行消息

点击后展开结构化详情，例如 target、session、correlationId、耗时、插件 ID、错误代码等。

要求：

- 颜色只用于辅助区分等级，不将整行染成强烈颜色。
- 普通消息使用项目常规字体；时间、代码、路径和技术值可用等宽字体。
- 支持复制单条日志和展开内容。
- 支持“仅查看同类日志”或等价的快速筛选，但不是硬性要求。
- 对相同错误短时间重复出现的情况，可显示聚合次数，避免刷屏。
- 列表应有明确的加载、空结果、错误和暂停状态。

### 4. 自动跟随

正确实现实时日志的滚动行为：

- 用户位于列表底部且自动跟随开启时，新日志自动滚到最新位置。
- 用户主动向上滚动阅读历史日志后，不得强制将其拉回底部。
- 此时继续接收日志，并显示“有 N 条新日志”之类的浮动入口。
- 点击入口后滚到最新并恢复跟随。
- “暂停”只暂停界面合并或刷新，不停止后端采集和文件写入。
- 恢复后应合并暂停期间的新日志，且不能造成长时间卡顿。

使用可靠的底部检测方式，例如 sentinel/IntersectionObserver 或带容差的距离判断。为相关行为编写测试。

### 5. 性能

- 前端不得无限累积日志对象。
- 优先使用后端分页和有界前端列表。
- 搜索可做轻量防抖。
- 实时日志批量合并，避免每条日志造成高频全页渲染。
- 在不新增大型依赖的前提下，使用窗口化、截断或分批加载等适合当前数据量的方案。
- 日志页面关闭后清理事件订阅和定时器。

### 6. 可访问性

- 所有图标按钮有可读标签。
- 键盘可以进入筛选、搜索、列表和菜单。
- 展开项具有正确的 `aria-expanded`。
- 删除确认框有正确的初始焦点和焦点返回。
- 状态变化使用适度的 `aria-live`，避免每条实时日志都打扰屏幕阅读器。
- 尊重项目已有的减少动态效果设置或系统偏好。

## 七、日志操作

### 1. 清空当前视图

“清空当前视图”仅清除当前页面已经加载和显示的日志，或设置一个本地可视起点：

- 不删除磁盘文件
- 不停止采集
- 不影响其他窗口或未来查询
- 可以立即执行，通常无需危险确认
- 后续新日志继续出现

名称和行为必须与“删除日志”明确区分。

### 2. 删除本地日志

提供真正删除历史日志的操作：

- 删除 7 天前日志
- 删除当前会话之前的日志
- 删除全部本地日志

可根据实际存储方案调整选项，但必须：

- 使用明确的危险确认
- 说明不会删除设置、插件、模型、电量历史或其他业务数据
- 处理文件正在写入的情况
- 删除后保持当前会话日志系统可继续工作
- 返回成功、部分成功或失败结果
- 在 Windows、macOS、Linux 上行为一致

### 3. 导出日志

至少支持：

- 导出当前筛选结果
- 导出本次运行会话日志
- 生成完整脱敏诊断包

普通日志建议使用 UTF-8 的 `.jsonl`，可另附人类可读 `.log`，但不要维护两套不一致的数据源。

导出必须：

- 使用系统保存对话框或项目现有文件导出模式
- 大数据由后端流式或直接写入文件，避免全部经 IPC 传到前端
- 导出前应用统一脱敏
- 文件名包含应用名、日期和会话或范围
- 对取消保存不显示为错误
- 对失败给出明确、可翻译的提示

### 4. 诊断包

整合并复用项目已有的脱敏诊断能力，不要继续维护相互独立的旧诊断导出和新日志导出流程。

诊断包可采用 ZIP，建议包含：

```text
mira-diagnostics-<date>.zip
├── summary.json
├── logs.jsonl
├── app-info.json
├── plugin-status.json
├── local-ai-status.json
└── privacy-report.txt
```

根据实际数据源调整文件，但必须满足：

- `summary.json` 提供版本、平台、架构、会话、错误/警告摘要
- 日志范围有限且明确
- 插件状态不包含秘密和原始设备通信
- 本地 AI 状态不包含完整样本历史
- `privacy-report.txt` 说明包含和明确不包含的内容
- 诊断包仅保存到用户选择的本地位置，不自动上传

现有关于页“隐私与数据”区域如果已有导出诊断按钮，应重用同一诊断服务。可以将其改为打开日志与诊断页或直接调用新的诊断包导出，但不得形成两套行为不一致的导出机制。

### 5. 其他操作

根据实现成本提供：

- 复制单条日志
- 复制当前筛选结果
- 打开日志目录
- 查看日志存储占用
- 开始临时诊断会话

其中“打开日志目录”只有在使用安全、跨平台且符合项目权限模型的实现时才加入。不要为了这个次要功能引入过宽 shell 权限。

## 八、临时诊断模式

实现用户主动触发的短时诊断模式，或至少为其建立完整基础能力。

建议行为：

1. 用户点击“开始临时诊断”。
2. 明确说明会临时记录更详细信息，但仍会脱敏且不会上传。
3. 将相关模块采集等级提高到 `debug`，必要时允许受控 `trace`。
4. 默认持续 5～15 分钟，可手动提前停止。
5. 到期后自动恢复原采集等级，即使日志页已关闭。
6. 结束后可直接导出本次诊断包。
7. 应用重启后不应意外永久保持 `trace`。

不要在诊断模式中取消所有隐私限制。详细程度提高不等于可以记录秘密、完整原始 HID 数据或无限样本。

## 九、关于页卡片

关于页第二块卡片应保持轻量，不在卡片内嵌入日志列表。

建议内容语义：

- 标题：日志与诊断
- 说明：查看程序、鼠标插件和本地 AI 的本地运行记录，用于排查设备连接、插件执行和预测问题
- 隐私提示：日志仅保存在本机
- 主按钮：打开日志与诊断

如果后端可以低成本提供状态，可显示最近错误/警告数量或日志占用；否则不要为了状态展示增加昂贵轮询。

确保该卡片在 DOM 与视觉顺序上确实是第二块。为此编写组件测试，避免未来回归。

## 十一、错误处理与可靠性

- 日志系统内部错误不能通过同一路径无限递归记录。
- 文件写入、序列化、事件发送和导出失败应有独立的降级路径。
- 锁中不得执行可能阻塞的长时间文件写入、压缩、IPC 或 UI 事件发送。
- 避免持有控制器锁后再次进入日志回调形成死锁。
- 子进程 stderr 读取线程必须可退出并回收。
- 导出和删除使用临时文件、原子替换或其他安全方式，避免生成半成品。
- 关闭应用时合理 flush，但不得造成明显退出卡顿。
- 时间、文件路径和轮转行为要考虑 Windows/macOS/Linux 差异。
- 对无法解析的旧日志文件应跳过并报告摘要，不能导致日志页完全打不开。
